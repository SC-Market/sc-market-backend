# Decoupled Inventory Model (V2.1)

## Problem

Currently, stock lots (`listing_item_lots`) are tightly coupled to listings via `listing_items.item_id`. This means:

- You can only track inventory that's attached to a listing
- Unlisted inventory doesn't exist in the system
- Moving stock between listings requires deleting and recreating lots
- There's no personal inventory view independent of the market
- Custom listings (not linked to a game item) need a sentinel `__custom_item__` hack

## Design

### Core Change

Stock lots belong to an **owner + game item + variant**, not a listing. Listings *reference* lots rather than *owning* them.

```
Current:
  listing_item_lots.item_id → listing_items (listing-scoped)

Proposed:
  stock_lots_v2.owner_id    → accounts (user-scoped)
  stock_lots_v2.game_item_id → game_items (nullable — for custom items)
  stock_lots_v2.variant_id  → item_variants
  stock_lots_v2.listing_id  → listings (nullable — not all stock is listed)
```

### Key Principle

The `game_item_id` on a lot is **auto-determined** when the lot is linked to a game-item-linked listing. If the listing isn't linked to a game item (custom listing), the `game_item_id` can be separately specified on the lot, or left NULL.

### Schema

#### `stock_lots_v2` (replaces `listing_item_lots`)

```sql
CREATE TABLE stock_lots_v2 (
  lot_id        UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  owner_id      UUID NOT NULL REFERENCES accounts(user_id),

  -- Game item link: auto-set from listing if listing has a game item,
  -- otherwise manually specified or NULL for pure custom items
  game_item_id  UUID REFERENCES game_items(id),
  variant_id    UUID REFERENCES item_variants(variant_id),

  -- Optional listing link: NULL means unlisted personal inventory
  listing_id    UUID REFERENCES listings(listing_id) ON DELETE SET NULL,

  quantity_total INTEGER NOT NULL CHECK (quantity_total >= 0),
  location_id   UUID REFERENCES locations(location_id) ON DELETE SET NULL,
  listed        BOOLEAN NOT NULL DEFAULT false,
  notes         VARCHAR(1000),
  crafted_by    UUID,
  crafted_at    TIMESTAMPTZ,
  created_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at    TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_stock_lots_v2_owner ON stock_lots_v2(owner_id);
CREATE INDEX idx_stock_lots_v2_game_item ON stock_lots_v2(game_item_id);
CREATE INDEX idx_stock_lots_v2_listing ON stock_lots_v2(listing_id) WHERE listing_id IS NOT NULL;
CREATE INDEX idx_stock_lots_v2_listed ON stock_lots_v2(listing_id, listed) WHERE listed = true;
```

#### Behavior Rules

1. **Linking a lot to a listing:**
   - If the listing has a `listing_items` row with a `game_item_id`, the lot's `game_item_id` is auto-set to match
   - If the listing is custom (no game item), the lot keeps whatever `game_item_id` it already has (or NULL)
   - `listed` must be `true` for the lot to count toward the listing's available quantity

2. **Unlinking a lot from a listing:**
   - Set `listing_id = NULL` and `listed = false`
   - `game_item_id` and `variant_id` are preserved (the lot remembers what it is)

3. **Creating a lot without a listing:**
   - User specifies `game_item_id` (optional) and `variant_id` (optional)
   - `listing_id = NULL`, `listed = false`
   - This is personal inventory tracking

4. **Listing quantity trigger:**
   - Same as current: sums `quantity_total` where `listing_id = X AND listed = true`
   - Updated to query `stock_lots_v2` instead of `listing_item_lots`

### What Changes

| Current (`listing_item_lots`) | New (`stock_lots_v2`) |
|---|---|
| `item_id` → `listing_items` (required) | `listing_id` → `listings` (nullable) |
| Game item determined by listing | `game_item_id` on lot directly (auto-set from listing or manual) |
| No `owner_id` (implied by listing seller) | `owner_id` required (explicit ownership) |
| `listed` controls purchase visibility | `listed` controls purchase visibility (same) |
| Lot deleted when listing deleted (`CASCADE`) | Lot unlinked when listing deleted (`SET NULL`) |

### What Stays the Same

- `variant_id` — still links to `item_variants` for quality/attribute tracking
- `location_id` — still links to `locations` table
- `quantity_total` — same semantics
- `listed` boolean — same semantics (controls whether lot counts toward listing quantity)
- `notes`, `crafted_by`, `crafted_at` — same
- `stock_allocations` — still links `lot_id` → `order_id` for order fulfillment
- Allocation service — same logic, just references `stock_lots_v2` instead of `listing_item_lots`

### API Changes

#### New: Personal Inventory Endpoints

```
GET    /api/v2/inventory              — list all lots for current user (across all listings + unlisted)
POST   /api/v2/inventory              — create unlisted lot (personal inventory)
PUT    /api/v2/inventory/:lot_id      — update lot (quantity, location, notes, game_item_id)
DELETE /api/v2/inventory/:lot_id      — delete lot
POST   /api/v2/inventory/:lot_id/list — link lot to a listing (sets listing_id + listed=true)
POST   /api/v2/inventory/:lot_id/unlist — unlink lot from listing (sets listing_id=NULL + listed=false)
```

#### Modified: Stock Lots Endpoints

Existing `/api/v2/stock-lots` endpoints continue to work but now operate on `stock_lots_v2`. Creating a lot via the listing-scoped endpoint auto-sets `listing_id` and `game_item_id`.

### Quantity Trigger Update

```sql
CREATE OR REPLACE FUNCTION update_listing_quantity()
RETURNS TRIGGER AS $$
DECLARE
  affected_listing_id UUID;
BEGIN
  -- Determine which listing was affected
  affected_listing_id := COALESCE(NEW.listing_id, OLD.listing_id);

  IF affected_listing_id IS NOT NULL THEN
    UPDATE listing_items
    SET
      quantity_available = (
        SELECT COALESCE(SUM(quantity_total), 0)
        FROM stock_lots_v2
        WHERE listing_id = affected_listing_id AND listed = true
      ),
      variant_count = (
        SELECT COUNT(DISTINCT variant_id)
        FROM stock_lots_v2
        WHERE listing_id = affected_listing_id AND listed = true
      )
    WHERE listing_id = affected_listing_id;
  END IF;

  -- If lot moved between listings, also update the old listing
  IF TG_OP = 'UPDATE' AND OLD.listing_id IS DISTINCT FROM NEW.listing_id AND OLD.listing_id IS NOT NULL THEN
    UPDATE listing_items
    SET
      quantity_available = (
        SELECT COALESCE(SUM(quantity_total), 0)
        FROM stock_lots_v2
        WHERE listing_id = OLD.listing_id AND listed = true
      ),
      variant_count = (
        SELECT COUNT(DISTINCT variant_id)
        FROM stock_lots_v2
        WHERE listing_id = OLD.listing_id AND listed = true
      )
    WHERE listing_id = OLD.listing_id;
  END IF;

  RETURN COALESCE(NEW, OLD);
END;
$$ LANGUAGE plpgsql;
```

### Migration Path

1. Create `stock_lots_v2` table
2. Copy data from `listing_item_lots`:
   ```sql
   INSERT INTO stock_lots_v2 (lot_id, owner_id, game_item_id, variant_id, listing_id, quantity_total, ...)
   SELECT
     lil.lot_id,
     COALESCE(l.seller_id, lil.owner_id) as owner_id,
     li.game_item_id,
     lil.variant_id,
     li.listing_id,
     lil.quantity_total,
     ...
   FROM listing_item_lots lil
   JOIN listing_items li ON lil.item_id = li.item_id
   JOIN listings l ON li.listing_id = l.listing_id
   ```
3. Update `stock_allocations` FK to reference `stock_lots_v2`
4. Update trigger to use `stock_lots_v2`
5. Update all services/controllers to use `stock_lots_v2`
6. Drop `listing_item_lots` (or rename to `_deprecated`)
