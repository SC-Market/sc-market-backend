# Decoupled Inventory Model (V2.1)

## Problem

Currently, stock lots (`listing_item_lots`) are tightly coupled to listings via `listing_items.item_id`. This means:

- You can only track inventory that's attached to a listing
- Unlisted inventory doesn't exist in the system
- Moving stock between listings requires deleting and recreating lots
- There's no personal inventory view independent of the market
- Custom listings (not linked to a game item) need a sentinel `__custom_item__` hack

## Design

### Core Change

Stock lots gain direct ownership and game item references. Listings become optional — lots can exist as personal inventory without being listed for sale.

### Approach: In-Place ALTER

Instead of creating a new table, we alter `listing_item_lots` to add the new columns. This preserves all existing data, FKs (`stock_allocations.lot_id`), and mapping tables (`v1_v2_stock_lot_map`).

### Schema Changes

```sql
-- 1. Add new columns (all nullable initially)
ALTER TABLE listing_item_lots
  ADD COLUMN owner_id UUID REFERENCES accounts(user_id),
  ADD COLUMN game_item_id UUID REFERENCES game_items(id),
  ADD COLUMN listing_id UUID REFERENCES listings(listing_id) ON DELETE SET NULL;

-- 2. Backfill from existing relationships
UPDATE listing_item_lots lil
SET
  owner_id = COALESCE(lil.owner_id, l.seller_id),
  game_item_id = li.game_item_id,
  listing_id = li.listing_id
FROM listing_items li
JOIN listings l ON li.listing_id = l.listing_id
WHERE lil.item_id = li.item_id;

-- 3. Make owner_id NOT NULL after backfill
ALTER TABLE listing_item_lots ALTER COLUMN owner_id SET NOT NULL;

-- 4. Make item_id nullable (lots no longer require a listing_item)
ALTER TABLE listing_item_lots ALTER COLUMN item_id DROP NOT NULL;

-- 5. Add indexes
CREATE INDEX idx_listing_item_lots_owner ON listing_item_lots(owner_id);
CREATE INDEX idx_listing_item_lots_game_item ON listing_item_lots(game_item_id);
CREATE INDEX idx_listing_item_lots_listing_id ON listing_item_lots(listing_id) WHERE listing_id IS NOT NULL;
```

### Column Summary After Migration

| Column | Required | Source | Notes |
|---|---|---|---|
| `lot_id` | NOT NULL | Existing PK | Unchanged |
| `item_id` | **nullable** | `listing_items` FK | Set when linked to a listing, NULL for unlisted inventory |
| `variant_id` | NOT NULL | Existing | Unchanged |
| `owner_id` | **NOT NULL** | **New** — backfilled from listing seller | Explicit ownership |
| `game_item_id` | nullable | **New** — backfilled from `listing_items` | Auto-set from listing if game-item-linked, manual otherwise |
| `listing_id` | nullable | **New** — backfilled from `listing_items` | NULL = personal inventory, set = listed for sale |
| `quantity_total` | NOT NULL | Existing | Unchanged |
| `location_id` | nullable | Existing | Unchanged |
| `listed` | NOT NULL | Existing | Unchanged — controls whether lot counts toward listing quantity |
| `notes` | nullable | Existing | Unchanged |
| `crafted_by` | nullable | Existing | Unchanged |
| `crafted_at` | nullable | Existing | Unchanged |

### Behavior Rules

1. **Linking a lot to a listing:**
   - Set `listing_id` and `item_id` (the listing_items FK)
   - `game_item_id` auto-set from `listing_items.game_item_id` if the listing has one
   - `listed = true` to count toward available quantity

2. **Unlinking a lot from a listing:**
   - Set `listing_id = NULL`, `item_id = NULL`, `listed = false`
   - `game_item_id` and `variant_id` preserved

3. **Creating unlisted inventory:**
   - `listing_id = NULL`, `item_id = NULL`, `listed = false`
   - `owner_id` required, `game_item_id` optional
   - This is personal inventory tracking

4. **Deleting a listing:**
   - `ON DELETE SET NULL` on `listing_id` — lots become unlisted inventory
   - `item_id` should also be NULLed (via trigger or application code)

### Quantity Trigger Update

```sql
CREATE OR REPLACE FUNCTION update_quantity_available()
RETURNS TRIGGER AS $$
DECLARE
  affected_item_id UUID;
  old_item_id UUID;
BEGIN
  affected_item_id := COALESCE(NEW.item_id, OLD.item_id);
  old_item_id := OLD.item_id;

  -- Update the listing_item that this lot belongs to
  IF affected_item_id IS NOT NULL THEN
    UPDATE listing_items
    SET
      quantity_available = (
        SELECT COALESCE(SUM(quantity_total), 0)
        FROM listing_item_lots
        WHERE item_id = affected_item_id AND listed = true
      ),
      variant_count = (
        SELECT COUNT(DISTINCT variant_id)
        FROM listing_item_lots
        WHERE item_id = affected_item_id AND listed = true
      )
    WHERE item_id = affected_item_id;
  END IF;

  -- If lot moved between listings, also update the old listing_item
  IF TG_OP = 'UPDATE' AND old_item_id IS DISTINCT FROM NEW.item_id AND old_item_id IS NOT NULL THEN
    UPDATE listing_items
    SET
      quantity_available = (
        SELECT COALESCE(SUM(quantity_total), 0)
        FROM listing_item_lots
        WHERE item_id = old_item_id AND listed = true
      ),
      variant_count = (
        SELECT COUNT(DISTINCT variant_id)
        FROM listing_item_lots
        WHERE item_id = old_item_id AND listed = true
      )
    WHERE item_id = old_item_id;
  END IF;

  RETURN COALESCE(NEW, OLD);
END;
$$ LANGUAGE plpgsql;
```

### API Changes

#### New: Inventory Endpoints

```
GET    /api/v2/inventory              — all lots for current user (listed + unlisted)
POST   /api/v2/inventory              — create unlisted lot (personal inventory)
PUT    /api/v2/inventory/:lot_id      — update lot
DELETE /api/v2/inventory/:lot_id      — delete lot
POST   /api/v2/inventory/:lot_id/list — link lot to a listing
POST   /api/v2/inventory/:lot_id/unlist — unlink lot from listing
```

#### Existing: Stock Lots Endpoints

`/api/v2/stock-lots` continues to work — these are listing-scoped views of the same `listing_item_lots` table, filtered by `listing_id IS NOT NULL`.

### What Stays the Same

- Table name: `listing_item_lots` (no rename needed)
- `stock_allocations.lot_id` FK — unchanged
- `v1_v2_stock_lot_map` — unchanged
- All existing queries that filter by `item_id` — still work for listed lots
- Allocation service — unchanged

### Frontend Changes

- Rename "Manage Stock" → "Manage Inventory" in sidebar
- Inventory page shows all lots grouped by game item (listed + unlisted)
- Lots can be linked/unlinked to listings from inventory view
- Stock manager on listing pages shows only lots linked to that listing

### Implementation Order

1. Knex migration: add columns, backfill, add constraints
2. Update quantity trigger to handle lot movement
3. Add inventory endpoints
4. Update stock lot service to support unlisted lots
5. Frontend: rename sidebar, add inventory page
6. Remove `__custom_item__` sentinel from migration service
