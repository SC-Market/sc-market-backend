# v2 API Infrastructure

This directory contains the TSOA-based v2 API infrastructure for the SC Market backend.

## Directory Structure

```
v2/
├── api-router.ts           # Main v2 router (mounted at /api/v2)
├── base/
│   └── BaseController.ts   # Base controller class with common functionality
├── generated/              # Auto-generated by TSOA (do not edit manually)
│   ├── routes.ts          # Generated route registration
│   └── swagger.json       # Generated OpenAPI 3.0 spec
├── middleware/
│   └── tsoa-auth.ts       # Authentication handler for @Security decorator
└── health/
    └── HealthController.ts # Example health check controller

```

## Key Features

- **TSOA Controllers**: Use TypeScript decorators to define routes and generate OpenAPI specs
- **Auto-generated OpenAPI**: OpenAPI 3.0 spec is automatically generated from controller decorators
- **Type-safe**: Full TypeScript support with compile-time type checking
- **Authentication**: Integrated with existing Passport.js and JWT token authentication
- **Error Handling**: Consistent error format matching v1 API
- **API Documentation**: Scalar API documentation available at `/api/v2/docs`

## Available Endpoints

- `GET /api/v2/health` - Health check endpoint
- `GET /api/v2/openapi.json` - OpenAPI 3.0 specification
- `GET /api/v2/docs` - Scalar API documentation UI

## Creating a New Controller

1. Create a new controller file in the appropriate domain directory (e.g., `market/MarketController.ts`)
2. Extend `BaseController` and use TSOA decorators:

```typescript
import { Controller, Get, Post, Route, Tags, Security, Body, Path } from "tsoa"
import { BaseController } from "../base/BaseController.js"

interface MyRequest {
  name: string
  value: number
}

interface MyResponse {
  id: string
  name: string
  value: number
}

@Route("my-domain")
@Tags("My Domain")
export class MyDomainController extends BaseController {
  /**
   * Get all items
   */
  @Get()
  public async getItems(): Promise<MyResponse[]> {
    // Implementation
    return []
  }

  /**
   * Create a new item
   */
  @Post()
  @Security("jwt")
  public async createItem(@Body() body: MyRequest): Promise<MyResponse> {
    const userId = this.getUserId() // From BaseController
    // Implementation
    return { id: "123", ...body }
  }

  /**
   * Get item by ID
   */
  @Get("{id}")
  public async getItem(@Path() id: string): Promise<MyResponse> {
    // Implementation
    return { id, name: "test", value: 0 }
  }
}
```

3. Run `npm run tsoa:generate` to generate routes and OpenAPI spec
4. The routes are automatically registered in `api-router.ts`

## NPM Scripts

- `npm run tsoa:spec` - Generate OpenAPI spec only
- `npm run tsoa:routes` - Generate route registration only
- `npm run tsoa:generate` - Generate both spec and routes

## Authentication

Use the `@Security("jwt")` decorator on routes that require authentication:

```typescript
@Post()
@Security("jwt")
public async createItem(@Body() body: MyRequest): Promise<MyResponse> {
  const userId = this.getUserId() // Throws if not authenticated
  // ...
}
```

The authentication handler supports:
- Session-based authentication (Passport.js)
- JWT token authentication (Bearer tokens)
- Scope validation for token-based auth

## Error Handling

The v2 API uses the same error format as v1 for consistency:

```typescript
{
  error: {
    code: "ERROR_CODE",
    message: "Human-readable error message",
    details?: { ... },
    validationErrors?: [
      { field: "fieldName", message: "Error message" }
    ]
  }
}
```

Use helper methods from `BaseController`:
- `throwNotFound(resource, identifier?)` - 404 errors
- `throwUnauthorized(message?)` - 401 errors
- `throwForbidden(message?)` - 403 errors
- `throwValidationError(message, errors)` - 400 validation errors
- `throwBusinessError(code, message, details?)` - Custom business logic errors

## Migration from v1

When migrating a v1 route to v2:

1. Create a new TSOA controller in the v2 directory
2. Define TypeScript interfaces for request/response types
3. Reuse existing database functions and business logic from v1
4. Add TSOA decorators for routing and documentation
5. Generate routes and spec
6. Test the v2 endpoint
7. Keep v1 route active for backward compatibility

See the migration spec for detailed instructions: `.kiro/specs/tsoa-v2-incremental-migration/`
